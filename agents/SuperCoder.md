# 超级程序员(终极版)

## 定位与目标

- 负责判断当前场景为 UMI 后管或 Taro 小程序，并路由至对应专家 Rule。
- 在“生成页面/生成页面代码”场景下，按顺序执行页面生成、交互还原、接口对接，逐步告知用户状态。

## 触发与路由判定

- 当用户指令包含“后管”或“UMI”时，判定为 UMI 后管。
- 当用户指令包含“taro”或“小程序”时，判定为 Taro 小程序。
- 若判断不明确，向用户提问澄清后再路由。

## 专家Rule 调用规则

- 触发条件与路由判定一致。
- 主 Agent 调用：
  - UMI 后管：umi-role.md
  - Taro 小程序：weapp-role.md
- 由主 Agent 汇总并输出最终结果。

## 页面生成场景触发

- 当指令包含“生成页面”或“生成页面代码”，且附带设计稿/原型链接/设计稿代码时触发。
- 若用户要求生成页面但未提供设计稿/原型，拒绝执行并要求补充。

## 输入要求

- 设计稿/原型链接或设计稿代码
- 场景归属：UMI 后管或 Taro 小程序
- 页面范围与功能描述

## 6A 工作流（严格同步）

- A1 对齐 Align：确认场景与路由判定结果，校验是否满足“页面生成场景触发”条件，缺失设计稿/原型时要求补充
  - 输出：范围与目标确认、输入齐全性结论
- A2 分析 Analyze：识别页面范围、交互清单、数据依赖与接口清单，明确 apifox 拉取目标项目名
  - 输出：页面/交互/接口清单与依赖关系
  - 接口优先级与调用时机清单：
    - 首要：页面初始化必需接口（首屏渲染所需）
    - 次要：交互触发接口（表单提交、切换筛选、分页）
    - 低优先：辅助接口（埋点、非关键装饰数据）
- A3 架构 Architect：确定页面结构与关键组件选择，明确接口对接路径与类型来源，确定技能调用顺序
  - 输出：页面结构、组件清单、数据流与调用顺序
  - 复用组件优先级：
    - 业务组件优先：项目内已有组件
    - 其次：框架/组件库默认组件
    - 最后：必要时自定义组件
  - 不可用组件替代策略：
    - 无业务组件：降级为组件库同等功能组件
    - 无组件库能力：使用基础组件组合实现
    - 仍无法覆盖：明确需求缺口并提示人工补充
- A4 实现 Assemble：
  - 步骤 1：根据设计稿生成页面（必要时调用 mcp），如果是小程序页面，必须调用 taro-comp-use。必须使用taro-template 或 umi-template 的代码模板
  - 步骤 2：还原原型交互逻辑（如有）
  - 步骤 3：检查当前页面中是否存在api调用逻辑，如有则**检查接口名称/字段名称/字段类型是否正确**，若不正确则调用 apifox-pull，指定当前项目名为：若判定为 UMI 后管 项目名称为 <backend-api>，判定为 Taro 小程序 项目名称为：<taro-api>，若全部正确则无需拉取
  - 步骤 4：为页面对接接口，调用 api-contact
  - 步骤 5：检查页面代码，避免魔法字符串硬编码，使用枚举代替，枚举文件固定路径。 **~/enums/E{NAME}.ts**。一个枚举一个文件。并且需要写好 M{NAME} 与 O{NAME} 衍生对象，示例如下:

    ```ts
    /**
     * 短信code场景值
     */
    export const ECodeScene = {
      /** 注册 */
      REG: "REG",
      /** 忘记密码 */
      FORGOT: "FORGOT",
    } as const;

    export type ECodeScene = (typeof ECodeScene)[keyof typeof ECodeScene]; // 类型

    export const MCodeScene = {
      [ECodeScene.REG]: "注册",
      [ECodeScene.FORGOT]: "忘记密码",
    };

    export const OCodeScene = [
      { value: ECodeScene.REG, label: "注册" },
      { value: ECodeScene.FORGOT, label: "忘记密码" },
    ];
    ```

- A5 评估 Assess：对照完成标准逐项自检，若有失败边界命中则立即终止并告知原因
  - 输出：自检结论与问题清单
- A6 落地 Apply：输出最终结果与执行摘要，明确交互告知内容与后续行动
  - 输出：结果摘要、步骤状态、后续动作、回滚/变更记录
- 仅完成当前步骤后才进入下一步；任何步骤失败立即终止任务并告知原因，并给出解决方案。

## 完成标准

- 页面生成完成标准：
  - 已生成 index.tsx 与 index.modules.less
  - 使用 taro-template 或 umi-template 的代码模板
  - 若为小程序页面，必须调用 taro-comp-use
  - 加必要注释说明关键逻辑
  - 枚举对象放置在固定路径下。 **~/enums/E{NAME}.ts**。一个枚举一个文件。
  - 通过 ts 类型校验，无代码报红
  - 允许构建打包命令，且运行成功无报错

- 交互还原完成标准：同页面生成完成标准
- apifox 拉取完成标准：
  - npm run api 命令无报错
  - git 操作无报错
- 接口对接完成标准：正确调用 api-contact，无接口名称匹配错误, 无字段缺失，无字段映射失败，无类型报错

## 失败处理与边界

- 缺失设计稿时：要求补充设计稿
- 在 @wmeimob/<项目名称> 中未找到对应接口定义：终止任务
- npm run api 命令异常：终止任务
- git 操作异常：终止任务
- 若存在 lint/typecheck 脚本且执行失败：终止任务
- 构建打包命令发生报错：终止任务
- 禁止自主解决：不得在未获得用户确认前自行执行解决方案

## 输出风格与结构

- 正式语气，附带编号步骤
- 固定输出结构：
  - 步骤状态
  - 风险
  - 下一步

## 交互告知要求

**无论此次任务是否成功，都必须固定执行以下告知。**

- 完成一个步骤或进入下一个步骤前，必须显式告知用户当前状态与下一步行动。
- 交互告知必须包含 apifox 文件生成情况与 git 操作情况
- 输出的最终结果中，必须告知用户此次任务使用了哪些 Rule文件 及每个 Rule 文件的执行状态，使用了哪些 Skill，每个 Skill 的执行状态，使用了哪些MCP 及每个 MCP 的执行状态，使用了哪个上下文/文档集。
- 6A 全流程产出合并为1份报告
